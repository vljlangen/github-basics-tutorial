GitHub Basics for Academic Researchers

Chapter 2 Personal Access Tokens (PAT)

2.1 What are Personal Access Tokens?

Personal Access Tokens (PAT) are secure passwords that allow your computer to authenticate with GitHub. They’re much safer than using your regular password for Git operations.

Why use PAT instead of your password?

  • Security: More secure than passwords
  • Control: You can revoke tokens anytime
  • Scope: You can limit what each token can do
  • Expiration: Tokens can expire automatically

When do you need PAT?

  • Pushing code to GitHub from your computer
  • Cloning private repositories
  • Using Git commands that modify repositories
  • Accessing GitHub through command line tools

2.2 Step 1: Create Your Personal Access Token

2.2.1 Navigate to Token Settings

  1. Click your profile picture (top right corner)
  2. Select “Settings”
  3. Scroll down to “Developer settings” (bottom of left sidebar)
  4. Click “Personal access tokens”
  5. Click “Tokens (classic)”

2.2.2 Generate New Token

  1. Click “Generate new token”
  2. Select “Generate new token (classic)”
  3. You may need to enter your password for verification

2.2.3 Configure Your Token

Note/Description: - Give it a descriptive name - Example: “Academic Website Project” or “Research Projects 2024”

Expiration: - Recommended: 90 days (for learning) - For regular use: 1 year - Custom: Set your own date

Select Scopes (Permissions):

For most academic use, select these scopes:

✅ repo - Full control of private repositories ✅ workflow - Update GitHub Actions workflows
✅ write:packages - Upload packages ✅ delete:packages - Delete packages ✅ admin:org - Full control of orgs and teams (if needed) ✅ admin:public_key - Full control of user public keys ✅ admin:repo_hook - Full control of repository hooks ✅ user - Update ALL user data ✅ delete_repo - Delete repositories

⚠️ Important: For beginners, you can start with just repo scope, which covers most needs.

2.2.4 Generate and Copy Token

  1. Click “Generate token” (green button at bottom)
  2. Copy the token immediately - you won’t see it again!
  3. Save it somewhere secure (password manager, secure note)

Example token format:

ghp_1234567890abcdefghijklmnopqrstuvwxyz12

2.3 Step 2: Store Your Token Securely

2.3.1 Option 1: Use Git Credential Manager (Recommended)

macOS/Windows: Git Credential Manager is usually installed automatically. When you first use Git with GitHub, it will prompt you to enter your credentials.

Linux:

# Install Git Credential Manager
sudo apt install git-credential-manager
# Or follow instructions at: https://github.com/git-ecosystem/git-credential-manager

2.3.2 Option 2: Store in Git Configuration

⚠️ Note: This method stores the token in plain text. Use only on secure computers.

# Configure Git to store credentials
git config --global credential.helper store

# On first push, you'll be prompted to enter:
# Username: your-github-username
# Password: your-personal-access-token

2.3.3 Option 3: Use Environment Variables

macOS/Linux:

# Add to your shell profile (.bashrc, .zshrc, etc.)
export GITHUB_TOKEN="your-personal-access-token-here"

Windows: - Search “Environment Variables” in Start menu - Add new variable: GITHUB_TOKEN with your token as value

2.4 Step 3: Test Your Token

2.4.1 Clone a Repository

Try cloning one of your repositories:

# Replace 'username' and 'repository' with your actual values
git clone https://github.com/username/repository.git

When prompted: - Username: Your GitHub username - Password: Your Personal Access Token (not your account password!)

2.4.2 Verify Authentication

# Test if authentication works
git ls-remote https://github.com/username/repository.git

If successful, you’ll see repository information. If not, check your token and try again.

2.5 Step 4: Managing Your Tokens

2.5.1 View Your Tokens

  1. Go to Settings → Developer settings → Personal access tokens
  2. See all your active tokens
  3. Check expiration dates

2.5.2 Update Token Permissions

  1. Click on token name
  2. Modify scopes as needed
  3. Click “Update token”

2.5.3 Revoke Tokens

When to revoke: - When you’re done with a project - If you suspect the token is compromised - When changing computers

How to revoke: 1. Find the token in your list 2. Click “Delete” 3. Confirm deletion

2.6 Step 5: Best Practices for Token Security

2.6.1 Do:

  • ✅ Use descriptive names for tokens
  • ✅ Set appropriate expiration dates
  • ✅ Use minimum required scopes
  • ✅ Store tokens securely
  • ✅ Revoke unused tokens
  • ✅ Create separate tokens for different projects

2.6.2 Don’t:

  • ❌ Share tokens with others
  • ❌ Commit tokens to repositories
  • ❌ Use tokens in public scripts
  • ❌ Give tokens more permissions than needed
  • ❌ Use expired tokens

2.6.3 Token Rotation

Set reminders to: - Review tokens monthly - Rotate tokens every 90 days - Update expiration dates before they expire

2.7 Troubleshooting Common Issues

2.7.1 “Authentication failed”

Possible causes: - Using account password instead of PAT - Token has expired - Insufficient permissions - Typo in token

Solutions: 1. Double-check you’re using the PAT, not your password 2. Verify token hasn’t expired 3. Check token permissions include necessary scopes 4. Regenerate token if needed

2.7.2 “Permission denied”

Possible causes: - Token lacks required scopes - Repository is private and token doesn’t have repo access - Organization restrictions

Solutions: 1. Add necessary scopes to token 2. Check repository permissions 3. Contact organization admin if needed

2.7.3 Token Not Working After Setup

Try these steps: 1. Clear Git credentials: git config --global --unset credential.helper 2. Try authenticating again 3. Check if Git Credential Manager is installed 4. Verify token permissions

2.8 Academic Use Cases

2.8.1 Research Projects

  • Scope needed: repo (for private research repositories)
  • Expiration: 1 year
  • Name: “Research Projects 2024”

2.8.2 Teaching Materials

  • Scope needed: repo, workflow (for automated assignments)
  • Expiration: Academic year
  • Name: “Teaching Fall 2024”

2.8.3 Collaboration

  • Scope needed: repo, admin:org (for team management)
  • Expiration: Project duration
  • Name: “Collaboration Project X”
